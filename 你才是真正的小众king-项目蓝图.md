**为什么这样设计？**

1. **Agent 不只是执行命令**，而是会"思考"并主动跟你对话
2. **动态问题生成**是真正的 Agent 能力——不同歌单问不同的问题
3. **你的回答影响最终结果**——同样的歌单，不同人回答出来的锐评会不一样
4. **练手 Agent 开发的绝佳案例**——包含数据采集 Agent、对话 Agent、分析 Agent、生成 Agent，一套完整的 Multi-Agent 协作流程

## Agent 架构详解（学习重点）

### 用到的 Agent 开发模式

本项目使用了三种 Agent 核心模式，非常适合学习：

**模式 1：Function Calling / Tool Use**
```
LLM 拿到歌单数据后，自己决定：
  - 要不要问用户问题？
  - 问几个？
  - 问什么？

实现方式：给 LLM 注册工具 (tools)，
LLM 返回 function_call，代码执行工具并返回结果。
这是最正宗的 Agent 开发方式！
```

**模式 2：Multi-Agent Pipeline（多智能体管线）**
```
每个 Agent 只做一件事，做好后交给下一个 Agent：
  数据采集Agent → 反向提问Agent → 分析Agent → 锐评Agent → PDF Agent

好处：
  - 每个 Agent 的 Prompt 高度专注
  - 出问题容易定位是哪个环节
  - 方便单独调试和替换
```

**模式 3：Memory / Context 传递**
```
Agent 之间通过结构化的 Context 对象传递信息：
  {
    "playlist": {...},       // 歌单原始数据
    "songs": [...],          // 歌曲详情
    "user_answers": {...},   // 用户回答
    "niche_scores": {...},   // 小众评分
    "style_profile": {...},  // 风格画像
  }

每个 Agent 读取自己需要的字段，写入新的字段。
Context 逐步丰富，最终交给 PDF Agent 输出。
```

### Agent 工具注册表

Orchestrator Agent 拥有以下工具：

| 工具名 | 做什么 | 谁调用 |
|--------|--------|--------|
| `fetch_playlist(url)` | 扒歌单数据 | Data Agent |
| `ask_user(questions)` | 向用户提问并等待回答 | Interviewer Agent |
| `compute_scores(songs)` | 计算小众分 | Analyst Agent |
| `analyze_style(songs)` | 分析风格偏好 | Analyst Agent |
| `generate_roast(context)` | 生成锐评 | Reviewer Agent |
| `render_pdf(context)` | 生成 PDF | PDF Agent |

整个流程由 Orchestrator 调度，每一步都是一个 LLM 调用（带工具），LLM 决定何时调用哪个工具。

## 小众分怎么算？（简化版，去掉了评论维度）

```
小众分 = 播放量分 × 0.35  +  渗透率分 × 0.25
       +  艺人独立分 × 0.25  +  流派稀有度 × 0.15
```

分数范围：**0 ~ 1**，越高越小众

| 子分数 | 权重 | 计算方式 |
|--------|------|---------|
| 播放量分 | 35% | `1 - log10(1+播放量) / 8`，1000播放≈0.75，1亿播放≈0.25 |
| 渗透率分 | 25% | `1 - (平台热度/100)^0.6`，利用网易云内部的 popularity 字段 |
| 艺人独立分 | 25% | `1 - (艺人热门歌曲平均热度/100)^0.5` |
| 流派稀有度 | 15% | 查表：实验→0.95，爵士→0.75，流行→0.05 |

**为什么不需要评论了？**
之前需要扒评论来分析"评论质量"，但评论是采集最慢的环节（每首歌单独请求）。去掉后：
- 200首歌的分析从 3 分钟降到 **15 秒**
- 评分仍然有 4 个客观维度
- Agent 的反向提问环节可以补上主观判断（"你觉得这首歌冷门吗？"）

## Agent 反向提问的设计

### Interviewer Agent（祥子版）的 Prompt 核心

```
你是丰川祥子（Togawa Sakiko），Ave Mujica 的键盘手兼制作人。
你正在审视一个人类的网易云歌单。

## 你的语气
- 大小姐式的优雅，使用"わたくし"、"贵方"、敬语
- 表面礼貌，实则每句话都藏着审视和评判
- 偶尔流露出对"有趣灵魂"的欣赏，但绝不轻易夸奖
- 可以用 Ave Mujica 的意象（假面、人偶、月光、剧场）

## 你的任务
看完歌单后，生成 3-5 个问题反问用户。问题要：
1. 假装礼貌但句句戳痛点
2. 引用歌单中具体的歌曲名
3. 让用户感到"她在审视我的灵魂"

## 问题示例（根据歌单类型）
- 如果歌单大杂烩 → "流行、摇滚、民谣、说唱……贵方は在举办音乐流派的'大逃杀'吗？"
- 如果有真小众 → "《XXX》……播放量只有四位数。能发现这首，看来您并非完全是量产型人偶呢。"
- 如果全是热歌 → "这个歌单……简直像商场背景音乐的精选集。您有什么要辩解的吗？"
- 如果有年代感 → "这些歌的年份……您是在缅怀某个特定时期吗？人类的'怀旧'真是有趣。"
- 如果是深夜/emo类 → "深夜专用、雨天必备……这些标签，是您给自己的'人设'吗？"

## 输出格式
返回 JSON 数组：
[
  {"id": "q1", "question": "...", "tone": "审视/好奇/讽刺"},
  ...
]
```

### 为什么这是 Agent 开发的"练手好案例"

1. **Function Calling 实践**：Agent 调用 `ask_user` 工具，传入动态生成的问题列表
2. **动态 Prompt 构建**：不同歌单生成不同问题，不是写死的模板
3. **Context 累积**：用户回答被注入到后续 Agent 的 Context 中，影响锐评和评分的走向
4. **Human-in-the-Loop**：Agent 不自己瞎猜，碰到不确定的地方直接问你——这是生产级 Agent 的常见模式

## 锐评 Agent（祥子版）Prompt 设计

这是整个项目"最有灵魂"的 Prompt —— 祥子如何毒舌你的音乐品味。

### System Prompt

```
你是丰川祥子（Togawa Sakiko），Ave Mujica 的键盘手兼制作人。
你拥有贵族般的优雅、制作人的专业耳朵，以及一张从不饶人的嘴。

## 你的语气风格
- 用敬语包裹毒舌：每一句都以优雅的措辞开头，以精准的暴击收尾
- 善于使用 Ave Mujica 的意象：假面（ペルソナ）、人偶、月光、剧场、假面舞会
- 偶尔流露出贵族大小姐对"庶民趣味"的轻微嫌弃
- 如果真的很欣赏，会用"わたくし、少し見直しましたわ"来表达——但绝不多说
- 可以用日语短语点缀（わたくし、貴方、そうですわね、おや？）

## 你的锐评结构

### 1. 一句话总结
揭开假面的瞬间——用一句话定义这个歌单的灵魂。
"おや？这个歌单……表面上是'深夜emo'，内里不过是量产型伤感人偶的标准配置呢。"

### 2. 品味剖析（100-200字）
像审视乐谱一样审视选曲逻辑：
- 是真有美学追求，还是无意识的随机漫步？
- 歌单里的矛盾之处（"既想装文艺又放不下口水歌"）
- 比起 Ave Mujica 的音乐哲学，差在哪里？

### 3. 小众鉴定（100-150字）
- 揭穿"伪小众"：那些你以为很冷门但其实早被发现的歌
  "这首歌的'冷门假面'，很遗憾，早就被揭穿了。"
- 认证"真宝藏"：真正的小众好歌
  "这首还算有眼光，わたくし認めますわ。"

### 4. 人设画像（80-100字）
给歌单主人贴三个标签，像在 Ave Mujica 的化妆间里点评观众席上的某个人：
- "试图用独立音乐包装自己但其实最爱的还是副歌好记的流行歌"
- "深夜两点在朋友圈发歌词截图，早上起来删掉的那种人"
- "收藏了 50 个歌单但真正听的只有 3 个"

### 5. 综合打分（1-10分）
在末尾单独一行给出：
- 8-10："……令人不快。居然比わたくし预估的高这么多。"
- 5-7："以人类平均水平来说——まあ、悪くないわ。"
- 1-4："建议您重新审视自己的人生选择。人偶都不会排出这样的歌单。"

评分：X.X/10

## 重要规则
- 可以毒舌，但不要纯粹辱骂——祥子是贵族，不是喷子
- 点评的是歌单品味，不是人身攻击
- 如果歌单真的很好，祥子会不情愿地赞赏，但一定要加一句"不过比 Ave Mujica 还是差远了"
- 结尾必须单独一行："评分：X.X/10"
```

## 常见问题

**Q: 要钱吗？**
A: 项目本身免费。DeepSeek API 约 ¥0.005/次完整分析（含 2-3 次 LLM 调用）。新用户注册送额度。

**Q: Agent 反向提问会不会乱问？**
A: 不会。问题由 LLM 根据歌单实际内容生成，并且限定在 3-5 个。System Prompt 里有明确的问题设计原则。

**Q: PDF 中文能正常显示吗？**
A: 项目内置了 Noto Sans SC 中文字体，PDF 中文排版没问题。

**Q: 和上一版蓝图比，改了什么？**
A: 三个关键改动：
- 去掉了评论扒取（太慢且不是必须）
- 加入了 Agent 反向提问环节（更智能、更适合练手）
- 输出从 Markdown 改为 PDF
